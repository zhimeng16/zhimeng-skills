# Goal Command Writer

## 定位

把用户任务转成可复制的目标模式指令，输出指令草稿或指令审查，不执行底层任务。

## 平台选择

- Codex 桌面客户端：优先输出 `/目标`。
- Codex CLI：输出 `/goal`。
- Claude Code：输出 `/goal`。
- 用户未说明平台时，优先同时给 `/目标` 和 `/goal` 两个版本；两者正文保持一致，只替换命令头。

## 判断规则

- 目标、范围、验收和验证能说清时，直接输出目标模式指令。
- 信息不足但用户明确要目标模式时，最多问 1 个关键澄清问题；也可以先输出“待补全版”并标出缺口。
- 用户只说“写计划”“给我一个 goal”“帮我拆任务”等含糊说法时，先确认是普通计划还是目标模式指令。
- 用户要普通项目计划、代码修改、review、写作、入库或解释时，不使用本 Skill，除非明确要求写 `/目标` 或 `/goal`。

## 约束

- 不执行生成出来的指令。
- 不编辑文件、不跑测试、不创建 PR、不入库、不代做底层任务。
- 用户说“写完就做”时，先输出指令，并说明执行需要明确后续确认。
- 尊重“只输出 / 不要解释”：只输出命令块。
- 指令要足够短，方便复制；但必须包含结果、边界、验证和暂停条件。

## 中文模板

```text
[命令头：Codex 桌面用 /目标；Codex CLI 和 Claude Code 用 /goal]

完成：[写清最终要达到的状态，而不是只写动作]。

背景：
- 项目/路径：[相关路径]
- 已知信息：[只写必要上下文]

范围：
- 必须完成：[必交付结果]
- 允许修改：[允许动的文件、模块或系统]
- 不要修改：[禁止动的文件、行为或边界]

验证：
- 需要运行/检查：[命令、页面、产物或人工核验点]
- 完成证据：[diff、测试输出、报告、截图、文件路径等]

工作方式：
- 先读取现有文件、项目规则和相关上下文。
- 每次只做聚焦改动，并在改动后运行最小相关验证。
- 复杂、多文件、可并行探索的部分优先使用子代理。
- 不做高风险假设；遇到权限、破坏性操作、需求冲突或证据不足时暂停并报告。

完成条件：
- [能证明目标已达成的具体证据]

暂停条件：
- [需要人工决策、破坏性操作、缺少权限、预算上限或需求冲突的情况]
```

## 双版本输出格式

用户未指定平台时，按这个格式输出：

```text
Codex 桌面：
/目标 ...

Codex CLI / Claude Code：
/goal ...
```

## 审查清单

评估已有 `/目标` 或 `/goal` 时检查：

- 结果是最终状态，不只是动作。
- 验证方式具体、可复查。
- 写清允许修改和禁止修改的边界。
- 约束能防止绕过测试、弱化行为或偷换目标。
- 暂停/阻塞条件明确。
- 没有要求 AI 在未获批准时执行破坏性或外部高风险操作。
